1. Current Architecture (v1)

The v1 pipeline has worked well for a small watchlist but has two hard dependencies that prevent horizontal scaling:

  │                                                                      │

  │    │                                                                 │
  │    ├── getBars()         → Alpaca Data API  (stock history)          │
  │    ├── getExpirations()  → localhost:4000   (API Gateway)  ◄─ DEP   │
  │    └── getOptionChain()  → localhost:4000   (API Gateway)  ◄─ DEP   │
  │                │                                                     │
  │                └── API Gateway (port 4000)                          │
  │                      ├── Alpaca Data API  (snapshots, no real OI)   │
  │                      ├── Alpaca Trade API (OI via /v2/options/...)   │
  │                      └── Yahoo Svc (port 5300) — Python/Flask/yfinance│
  │                                                                     │
  │  push-to-r2.js  →  Cloudflare R2  →  newleaf-alpaca-r2 (Firebase)  │
  └──────────────────────────────────────────────────────────────────────┘
⚠ v1 Problems Requires API Gateway (port 4000) and Python Yahoo service (port 5300) to both be running on the same machine. Cannot be deployed to a remote server or run as multiple instances. DTE filter bug (time-of-day rounding) caused 0-DTE expiries to be silently dropped. All 10 symbols run sequentially — too slow for 100+ stocks.

3. v2 Pipeline Design

3.1 Data Sources

Two APIs are used, each providing what the other lacks:

🔶 Alpaca Markets

  • Live bid / ask / mid prices
  • Greeks (gamma, delta, theta, vega) — OPRA feed
  • IV (implied volatility)
  • Volume (intraday)
  • Stock snapshot (price, OHLC, volume)
  • Stock bars (OHLCV history for technicals)

Base: data.alpaca.markets · Auth: API key header

🔵 Yahoo Finance (unofficial)

  • Open Interest per strike per expiry
  • Expiry date list
  • Strike list for each expiry
  • Volume cross-check

Direct HTTP — no API key needed · ~15 min delay on OI · Called via query2.finance.yahoo.com

⚠ Why Yahoo for OI? Alpaca's /v1beta1/options/snapshots data API returns openInterest: 0 for all contracts. Real OI is only available via api.alpaca.markets/v2/options/contracts (trading API) which requires a trading account auth. Yahoo Finance is the cleanest zero-auth alternative.

3.2 Fetch Strategy Per Symbol

1
Stock snapshot (Alpaca) Fetch current price, OHLC, volume, change% from /v2/stocks/{symbol}/snapshot.
2
Price bars for technicals (Alpaca) Fetch 160 days of daily OHLCV from /v2/stocks/{symbol}/bars. Compute: RSI-14, SMA-50, SMA-100, Bollinger Bands, support/resistance.
3
Expiry list + OI (Yahoo Finance) Call query2.finance.yahoo.com/v7/finance/options/{symbol} with no expiry param — returns list of all expiry timestamps and the ATM chain. Then for each relevant expiry, call with ?date={timestamp} to get full chain with real OI per strike. This is the only place OI comes from.
4
Live prices + Greeks (Alpaca) For each relevant expiry within DTE range, call /v1beta1/options/snapshots/{symbol}?expiration_date=... to get live bid/ask and Greeks (gamma, delta, IV) per contract. Note: Greeks only available on OPRA subscription; falls back to Black-Scholes estimate if null.
5
Merge on (strike, type, expiry) Join Yahoo OI into Alpaca snapshot contracts. Result: each contract has live price, Greeks, AND real OI.
6
Gamma wall analysis Compute GEX = gamma × OI × 100 × spot per strike. Find call wall (max call GEX above spot), put wall (max put GEX below spot), gamma flip. If gamma = 0 (no OPRA), fall back to OI-weighted wall detection.
7
Score + strategy Compute opportunity score (0–100) across Gamma / IV / Trend pillars. Select strategy: Iron Condor / Iron Butterfly / Directional Spreads.
8
Write JSON → R2 Upload reports/{SYMBOL}/latest.json and reports/{SYMBOL}/{date}.json. Update reports/manifest.json atomically.

5. Multi-Instance Architecture

The key design principle: R2 is the coordination layer. Each instance owns its own partition of the symbol list and writes independently. No inter-instance communication needed.

  ┌────────────────────────────────────────────────────────────────────────────┐
  │  Symbol List: 300 stocks                                                   │
  │  SPY QQQ MSFT AAPL NVDA TSLA AMZN GOOGL META AVGO ... (290 more)          │
  │                                                                            │
  │  Instance A (Mac Mini M4)      Instance B (VPS-1)    Instance C (VPS-2)   │
  │  Symbols: 1–100                Symbols: 101–200      Symbols: 201–300      │
  │  --shard=0 --total-shards=3    --shard=1 --shards=3  --shard=2 --shards=3 │
  │                                                                            │
  │  Each instance runs every 15 minutes (cron) independently                 │
  │  Concurrency: 5 symbols in parallel per instance                          │
  │  ~100 symbols × ~8s per symbol / 5 parallel = ~160s = under 3 min        │
  │                                                                            │
  │              ↓ All write to same R2 bucket ↓                              │
  │                                                                            │
  │  Cloudflare R2: newleaf bucket                                             │
  │    reports/SPY/latest.json        reports/GOOGL/latest.json               │
  │    reports/QQQ/latest.json        reports/META/latest.json                │
  │    ...                            reports/manifest.json  (merged)         │
  │                                                                            │
  │  Manifest strategy: each instance reads manifest, upserts its symbols,    │
  │  writes back. Last-write-wins per symbol key (safe, idempotent).          │
  └────────────────────────────────────────────────────────────────────────────┘
📐 Capacity calculation Yahoo Finance: ~2s per symbol (2 requests: expiries + chain per expiry) · Alpaca: ~1.5s per symbol (snapshot + bars + chain) · Total per symbol: ~4–6s · 5 parallel workers · 100 symbols = ~100s per run = well within 15 min window. For 300 symbols across 3 instances: each handles 100 symbols ≈ 100s per run.

Shard Assignment

  # Sorted symbol list partitioned by index modulo total shards
  # Ensures stable, even distribution regardless of list order

  symbols = sorted(ALL_SYMBOLS)
  my_symbols = [s for i, s in enumerate(symbols) if i % TOTAL_SHARDS == MY_SHARD]

  # Example: 9 symbols, 3 shards
  # Shard 0: AAPL, GOOGL, MSFT     (indices 0, 3, 6)
  # Shard 1: AMZN, META, NVDA      (indices 1, 4, 7)
  # Shard 2: AVGO, QQQ, SPY        (indices 2, 5, 8)

6. Script Architecture

The v2 pipeline is a single Node.js file with zero npm dependencies. Uses only Node 18+ built-ins (fetch, fs, path, crypto).

  newleaf-pipeline.js  (~700 lines, single file)
  ├── CONFIG LOADER          loadConfig(path) → reads JSON config file
  │
  ├── ALPACA CLIENT          (inline, no class)
  │   ├── alpacaGet(url)     → fetch with Alpaca auth headers
  │   ├── getSnapshot(sym)   → stock price, OHLC, volume
  │   ├── getBars(sym)       → 160d OHLCV for technicals
  │   └── getAlpacaChain()   → snapshots for expiry (Greeks + live prices)
  │
  ├── YAHOO CLIENT           (inline, no dependencies)
  │   ├── yahooGet(url)      → fetch with Yahoo headers + user-agent
  │   ├── getYahooExpiries() → list of expiry unix timestamps
  │   └── getYahooChain()    → full chain with real OI per strike
  │
  ├── TECHNICALS             (pure functions, identical to v1)
  │   ├── calcSMA()
  │   ├── calcBB()
  │   ├── calcRSI()
  │   └── analyzeTechnicals()
  │
  ├── GAMMA ANALYSIS         (pure functions, improved from v1)
  │   ├── estimateGamma()    → Black-Scholes fallback
  │   ├── mergeChains()      → join Yahoo OI + Alpaca Greeks by (strike,type,expiry)
  │   └── analyzeGamma()     → walls, flip, confidence, topStrikes
  │
  ├── SCORING                (pure functions, identical to v1)
  │   ├── calcScore()
  │   ├── getDirection()
  │   └── selectStrategy()
  │
  ├── R2 UPLOADER            (inline, uses @aws-sdk/client-s3 OR native fetch + signed URL)
  │   └── uploadToR2()
  │
  ├── MANIFEST MANAGER
  │   └── upsertManifest()   → read-modify-write per symbol
  │
  ├── processSymbol(sym)     → orchestrates all above for one symbol
  │
  └── main()                 → sharding, concurrency, CLI args, summary

Concurrency Model

  // Process symbols in batches of CONCURRENCY (default 5)
  for (let i = 0; i < symbols.length; i += CONCURRENCY) {
    const batch = symbols.slice(i, i + CONCURRENCY);
    await Promise.all(batch.map(sym => processSymbol(sym).catch(err => {
      results.push({ sym, ok: false, error: err.message });
    })));
  }
  // Each symbol's Yahoo + Alpaca calls run in parallel WITHIN processSymbol
  const [yahooChain, alpacaSnap, bars] = await Promise.all([
    getYahooData(sym),
    getAlpacaSnapshot(sym),
    getAlpacaBars(sym)
  ]);

8. Deployment Guide

Single instance (Mac Mini — current setup)

  # crontab -e
  */15 9-16 * * 1-5  cd /path/to/pipeline && node newleaf-pipeline.js --watchlist

  # Or with PM2 (restarts on crash):
  pm2 start newleaf-pipeline.js --name newleaf-pipeline --cron "*/15 9-16 * * 1-5"

3-instance setup (Mac Mini + 2× VPS)

  # Each server has:
  #   newleaf-pipeline.js
  #   config.json  (same Alpaca + R2 credentials, same full watchlist)

  # Mac Mini (shard 0 of 3):
  node newleaf-pipeline.js --watchlist --shard=0 --total-shards=3

  # VPS-1 (shard 1 of 3):
  node newleaf-pipeline.js --watchlist --shard=1 --total-shards=3

  # VPS-2 (shard 2 of 3):
  node newleaf-pipeline.js --watchlist --shard=2 --total-shards=3

Watching 300 stocks (full S&P-100 + extras)

  // Each shard picks every 3rd symbol alphabetically
  // All 3 instances finish in ~160s, all under 15-min cron window

✅ Zero coordination required Each instance reads from its own config, fetches its own slice of symbols, and writes to R2 independently. No shared state, no locking, no message queue. R2's last-write-wins is safe because each symbol's data is fully self-contained and instances never overlap (sharding is deterministic).

9. Known Limitations & Mitigations

LimitationImpactMitigation
Yahoo Finance rate limiting 429 errors during rapid scans 200ms jitter between Yahoo calls; exponential backoff; symbol-level catch so one failure doesn't stop batch
Yahoo OI is ~15-20 min delayed OI walls not perfectly real-time Acceptable for 15-min scan cadence; document clearly in UI
Alpaca Greeks require OPRA subscription gamma=0, falls back to Black-Scholes B-S estimate is used; chart shows OI bars instead of GEX bars; confidence score is penalised
Yahoo API is unofficial / undocumented Could break without notice Fallback path: if Yahoo fails, use Alpaca OI estimate + flag oi_source: "estimated" in report
R2 manifest write conflicts Manifest corruption if 2 instances write simultaneously Each instance reads full manifest, upserts only its own symbols, writes back. Worst case: one instance's data overwrites another's for the same run — benign since next run corrects it
0-DTE expiries expire at market close Chain data disappears mid-day DTE_MIN=0; expired contracts return empty chain which pipeline handles gracefully
No real-time streaming Data is 15-min snapshots By design for this use-case; real-time would require WebSocket subscriptions (future v3)

Yahoo Finance Direct API

The v2 pipeline calls Yahoo Finance directly (no Python service) using the same undocumented JSON endpoint that yfinance uses:

  # Expiries + ATM chain:
  GET https://query2.finance.yahoo.com/v7/finance/options/{SYMBOL}
    headers: { 'User-Agent': 'Mozilla/5.0', 'Accept': 'application/json' }
    → { optionChain: { result: [{ expirationDates: [unix...], options: [{calls,puts}] }] } }

  # Specific expiry:
  GET https://query2.finance.yahoo.com/v7/finance/options/{SYMBOL}?date={unix_timestamp}
    → { optionChain: { result: [{ options: [{ calls: [{strike,openInterest,...}], puts: [...] }] }] } }

  # Each call/put has: strike, openInterest, volume, impliedVolatility, bid, ask, lastPrice
